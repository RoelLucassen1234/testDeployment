Inleiding

In dit document zullen de volgende onderwerpen worden besproken over de applicatie: • Generale beschrijving met architectuur en user groups • Security requirements, data assets , CIAP

Wat is de applicatie

De applicatie is een reader website applicatie waarin mensen hun verhalen kunnen plaatsen. Deze verhalen kunnen persoonlijk zijn tot fictie met een daarbij horende comment sectie waarin mensen hun mening kunnen geven en discussies kunnen houden.

Wat is het doel van de applicatie

Het doel is om een veilige applicatie te maken die aan de beveiliging voldoet en daarnaast een openbare omgeving geeft om je mening etc te geven. Ook zal deze applicatie worden gebruikt voor red en blue teamers om hun vaardigheden te testen.

Architectuur

In dit kopje zal alle architectuur staan van het project. Het is mogelijk dat sommige architectuur mist omdat het nog niet is toegevoegd of gemaakt.

Systeem Context

Aangezien de applicatie een website is heeft een gebruiker alleen toegang tot de front-end. Vanuit de front-end kan hij bepaalde requests maken naar de backend/API. De API zal verifiëren dat de gebruiker de juiste privileges heeft en zal gebasseerd hierop een request maken naar een kleine database. Hiernaast zal hij alle transacties loggen. De database zal opnieuw verifiëren of de API request de juiste privileges heeft en zal gebasseerd hierop de request uitvoeren of niet.

Netwerk architectuur

Mijn netwerk architectuur zal er als volgt: - Firewall 1.0. Deze firewall restrict alle inbound en outbound communicatie. Een voorbeeld zou zijn om alleen port 80 en 443 verkeer open te zetten voor http en https servers. - IDS. Dit is een intrusion detection server die alle kritische subnets bekijkt en de LAN. Dit helpt met het vinden van een afwijkende netwerk process dat mogelijk een aanval kan zijn. - Webserver. Hierbinnen wordt de website gehost. - Application Server. Hier worden alle transacties van de webserver gelogged en geverifierd. Als de requests zijn goedgekeurd kan het zijn dat de application server een request naar de database stuurt - Firewall 2.0. Sluit alles behalve de ipadressen waar servers op staan gehost zoals de database. - Database. De database zal requests van de Applicatie Server binnenkrijgen, verifiëren en dan data terugsturen naar de application server

Security Requirement

De security is belangrijk in deze applicatie omdat er gebruik wordt gemaakt van emails en wachtwoorden. Dit is gevoelige informatie dat niet iedereen zomaar mag weten. Daarnaast is security ook belangrijk om ervoor te zorgen dat vreemde niet zomaar verhalen van andere kunnen aanpassen en die schrijvers een slechte reputatie geven. Dezelfde regel geld ook voor de reacties/comments van andere en/of het verwijderen ervan. Hetgene waar eerst op wordt gefocused is het authorizeren en authorisiatie. Een lezer heeft minder rechten dan een schrijver terwijl een schrijver minder rechten heeft dan een administrator. Daarnaast zal ook moeten worden gekeken hoe iemand zichzelf kan authentiseren. Dit kan doormiddel van two-factor authenticatie maar daar moet nog naar worden gekeken. Het tweede deel waar de applicatie zich op zal focussen is het tegengaan van XSS en brute-forcen van accounts. Het brute-forcen van accounts zal moeten worden gedaan door een time-out of captcha. Voor het probleem van de XSS zal nog onderzoek naar moeten worden gedaan.
